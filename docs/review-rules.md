# 📌 PR Review Rules

## 1. 목적

이 문서는 본 프로젝트의 Pull Request 리뷰 시 **일관된 기준을 적용하기 위한 규칙**을 정의한다.

리뷰는 단순 코드 스타일이 아니라 다음을 보장하는 것을 목표로 한다:

* 설계 문서와의 일관성 유지
* 서비스/도메인 책임 분리 준수
* 안정적인 운영을 위한 품질 확보
* 협업 규칙(브랜치/PR/테스트) 준수

---

## 2. 리뷰 필수 체크 항목

리뷰어 및 자동 리뷰 도구(Codex)는 아래 항목을 반드시 확인해야 한다.

---

### 2.1 PR 제목 규칙

#### ✅ 형식

```
[type](service): description
```

#### ✅ type 종류

| type     | 설명            |
| -------- | ------------- |
| feat     | 새로운 기능        |
| fix      | 버그 수정         |
| chore    | 설정, 빌드, 기타 변경 |
| refactor | 리팩토링          |
| test     | 테스트 코드        |
| docs     | 문서 변경         |

#### ✅ service

* 변경이 발생한 서비스명을 명시
* 예: gateway, member, event, commerce, payment, settlement, admin

#### ✅ 예시

```
feat(member): 회원가입 API 구현
fix(payment): 결제 승인 시 중복 요청 방지
chore(gateway): Swagger 설정 추가
```

#### ❌ 잘못된 예

```
회원가입 구현
feat: 회원가입
feat(member)
```

#### 🔍 리뷰 체크

* [ ] type이 명확한가
* [ ] 서비스명이 포함되어 있는가
* [ ] 변경 내용을 설명하고 있는가

---

### 2.2 브랜치 네이밍 규칙

#### ✅ 형식

```
{type}/{service}-{feature}
```

#### ✅ 예시

```
feat/member-signup
feat/event-create
feat/commerce-cart-add
feat/payment-pg-integration
feat/settlement-auto-process
fix/payment-duplicate-request
chore/gateway-swagger
```

#### ❌ 잘못된 예

```
signup
feature1
dev
test-branch
```

#### 🔍 리뷰 체크

* [ ] type이 포함되어 있는가
* [ ] 서비스명이 포함되어 있는가
* [ ] 기능 단위로 명확하게 표현되었는가

---

### 2.3 테스트 코드 작성 여부

#### ✅ 필수 원칙

* 모든 핵심 비즈니스 로직에는 테스트 코드가 포함되어야 한다
* 테스트 없이 PR을 머지하는 것은 원칙적으로 금지한다

---

#### ✅ 반드시 테스트해야 하는 영역

* Service 레이어 (비즈니스 로직)
* 결제 / 환불 / 정산 관련 로직
* 상태 변경 로직 (Order, Payment, Refund 등)
* 예외 발생 케이스

---

#### ✅ 테스트 내용 기준

* 정상 케이스 (Success)
* 실패 케이스 (Exception)
* 경계값 케이스 (Edge Case)

---

#### ✅ 예시

* 결제 승인 성공 테스트
* 중복 결제 요청 방지 테스트
* 환불 시 상태 전이 테스트
* 잔액 부족 예외 테스트

---

#### ❌ 잘못된 경우

* 테스트 코드 없음
* Happy Path만 존재
* 검증(assert) 없이 실행만 하는 테스트

---

#### 🔍 리뷰 체크

* [ ] 테스트 코드가 존재하는가
* [ ] 핵심 로직이 테스트되고 있는가
* [ ] 성공/실패 케이스가 모두 포함되어 있는가
* [ ] 테스트가 의미 있는 검증을 수행하는가

---

## 3. 리뷰 결과 작성 형식

리뷰는 아래 구조로 작성한다.

### 1. 변경 요약

* 어떤 기능/수정인지 간단히 설명

### 2. 문제점 (필수 수정)

* 규칙 위반 사항
* 버그 가능성
* 설계 위반

### 3. 개선 제안 (권장)

* 구조 개선
* 가독성 개선
* 테스트 보완

### 4. 테스트 관점

* 테스트 부족 여부
* 추가해야 할 테스트 제안

---

## 4. 머지 기준

다음 조건을 만족해야 머지 가능하다:

* [ ] PR 제목 규칙 준수
* [ ] 브랜치 네이밍 규칙 준수
* [ ] 테스트 코드 포함
* [ ] 핵심 로직 테스트 완료
* [ ] 치명적인 리뷰 코멘트 해결 완료

---

## 5. 자동 리뷰 도구(Codex) 적용 기준

Codex는 본 문서를 기준으로 PR을 리뷰해야 한다.

### Codex는 반드시 다음을 확인한다:

1. PR 제목 규칙 위반 여부
2. 브랜치 네이밍 규칙 위반 여부
3. 테스트 코드 존재 여부
4. 테스트 범위 적절성

### Codex 리뷰 출력 형식

```
[요약]
...

[문제점]
...

[개선 제안]
...

[테스트]
...
```

---

## 6. 금지 사항

* 규칙을 무시한 PR 머지
* 테스트 없는 핵심 로직 머지
* 의미 없는 PR 제목 작성
* 브랜치 목적이 불분명한 상태에서 작업
